# SmartFilteredHopper 容量补丁技术方案

## 1. 问题背景

当 `HopperCapacity > 36` 时，SmartFilteredHopper 的 `GetActualCapacity` Postfix 只修改了返回值，但存在以下问题：

| 问题 | 影响 |
|------|------|
| `ItemGrabMenu` 构造时硬编码 9列×4行=36格 | `SetupBorderNeighbors()` 可能因列数不匹配崩溃 |
| UI 不会随容量扩展 | 超出 36 格的物品无法通过 UI 访问 |
| 无滚动支持 | 超过 36 格的物品实际存在但不可见 |

## 4. 推荐方案：方案 A（简单方案）

### 4.1 核心思路

1. **保留**现有的 `GetActualCapacity` Postfix
2. **添加** `ItemGrabMenu` 构造 Prefix + Finalizer — 容量抑制模式，防止 `SetupBorderNeighbors` 崩溃
3. **添加** 物品转移 Prefix — 防止菜单重建闪烁
4. **不实现滚动** — 最大容量限制为 **72 格**（6行×12列）
5. **配置限制** — `HopperCapacity` 最大值设为 72

### 4.2 需实现的补丁

| # | 目标方法 | 类型 | 作用 |
|---|----------|------|------|
| 1 | `Chest.GetActualCapacity` | Postfix | 已有，保留 |
| 2 | `ItemGrabMenu` 构造 | Prefix + Finalizer | 容量抑制 + 防止崩溃 |
| 3 | `Chest.grabItemFromChest` | Prefix | 跳过 ShowMenu() 重建 |
| 4 | `Chest.grabItemFromInventory` | Prefix | 同上 |

### 4.3 容量抑制模式（关键）

```csharp
[ThreadStatic]
private static bool _suppressCapacity;

private static void ItemGrabMenuCtor_Prefix(Item sourceItem) {
    // 检查是否是 hopper chest
    bool isOurs = sourceItem is Chest c && c.modData.ContainsKey(CapacityModDataKey);
    if (isOurs)
        _suppressCapacity = true;
}

private static Exception ItemGrabMenuCtor_Finalizer(ItemGrabMenu __instance, Exception __exception, Item sourceItem) {
    bool wasSuppressed = _suppressCapacity;
    _suppressCapacity = false;

    if (__exception != null) return __exception;
    if (!wasSuppressed) return null;

    // Finalizer 中不做 ResizeMenu，简单清空标记即可
    // 菜单按 vanilla 36 格显示，超出物品无法通过 UI 访问
    return null;
}
```

**原理**：`SetupBorderNeighbors()` 在 `ItemGrabMenu` 构造时基于 36 格布局计算邻居关系。当容量大于 36 时，直接构造会导致列数不匹配崩溃。通过抑制模式，让构造按 vanilla 36 格完成，然后再恢复正常容量。

### 4.4 物品转移前缀

```csharp
private static bool GrabItemFromChest_Prefix(Chest __instance, Item item, Farmer who) {
    if (Game1.activeClickableMenu is not ItemGrabMenu igm)
        return true;

    // 检查是否是 hopper chest
    var foundChest = FindHopperChest(igm);
    if (foundChest != __instance) return true;

    if (who.couldInventoryAcceptThisItem(item)) {
        __instance.GetItemsForPlayer().Remove(item);
        __instance.clearNulls();
    }
    return false;  // 跳过原方法，防止 ShowMenu() 重建
}

private static bool GrabItemFromInventory_Prefix(Chest __instance, Item item, Farmer who) {
    if (Game1.activeClickableMenu is not ItemGrabMenu igm)
        return true;

    var foundChest = FindHopperChest(igm);
    if (foundChest != __instance) return true;

    // 保留 snapped component
    int snappedId = igm.currentlySnappedComponent?.myID ?? -1;

    // 复制 vanilla 逻辑但不调用 ShowMenu()
    if (item.Stack == 0) item.Stack = 1;
    Item remainder = __instance.addItem(item);
    if (remainder == null)
        who.removeItemFromInventory(item);
    else
        remainder = who.addItemToInventory(remainder);

    __instance.clearNulls();
    igm.heldItem = remainder;

    if (snappedId != -1) {
        igm.currentlySnappedComponent = igm.getComponentWithID(snappedId);
        igm.snapCursorToCurrentSnappedComponent();
    }

    return false;
}
```

### 4.5 FindHopperChest 查找逻辑

```csharp
private static Chest FindHopperChest(ItemGrabMenu menu) {
    // 1. 检查 sourceItem 是否是带标记的 Chest
    if (menu.sourceItem is Chest srcChest
        && srcChest.modData.ContainsKey(CapacityModDataKey)) {
        return srcChest;
    }

    // 2. 检查 menu.context 是否是 hopper
    if (menu.context is StardewValley.Object obj
        && Utill.TryExtractHopper(obj, out var hopper)) {
        return hopper;
    }

    // 3. 通过 Game1.currentLocation 遍历查找 heldObject 是该 chest 的对象
    if (Game1.currentLocation == null) return null;

    foreach (var locObj in Game1.currentLocation.Objects.Values) {
        if (Utill.TryExtractHopper(locObj, out var hopper)
            && hopper.heldObject.Value is Chest chest
            && ReferenceEquals(chest.Items, menu.ItemsToGrabMenu?.actualInventory)) {
            return chest;
        }
    }

    return null;
}
```

## 7. 其他 Mod 冲突风险分析

### 7.1 补丁失效的常见原因

| 原因 | 场景示例 | 影响 |
|------|----------|------|
| **目标方法签名改变** | 游戏更新或其他 mod 重写了目标方法 | 补丁匹配失败，完全失效 |
| **方法内联（Inlining）** | JIT 编译器将小方法内联，补丁无法拦截 | 补丁不执行 |
| **执行顺序问题** | 其他 mod 后 patch 先执行，覆盖了我们的修改 | 功能被覆盖 |
| **Transpiler 冲突** | 两个 mod 都修改同一方法的 IL 指令 | 指令顺序不确定，可能崩溃 |
| **状态污染** | ThreadStatic 或静态字段被其他 mod 修改 | 容量抑制模式失效 |

### 7.2 可能与本补丁冲突的 Mod 类型

| Mod 类型 | 冲突风险 | 典型 Mod |
|----------|----------|----------|
| **UI 扩容 Mod** | 高 — 同样 patch `ItemGrabMenu` | BiggerAutoGrabber, BetterChests |
| **箱子功能 Mod** | 中 — 可能 patch `Chest.grabItemFromChest/Inventory` | ChestsAnywhere, InfiniteStorage |
| **菜单美化 Mod** | 中 — 可能 patch `ItemGrabMenu.draw` | UIInfoSuite, BetterCrafting |
| **Automate 生态** | 低 — patch `Machine` 相关而非 chest | Automate |

### 7.3 具体冲突场景

#### 场景 1: BiggerAutoGrabber 同时启用

**问题**：`GetActualCapacity` 被两个 mod 同时 postfix

```csharp
// SmartFilteredHopper patch
__result = cap;  // 可能被 BAG 覆盖

// BiggerAutoGrabber patch
__result = cap;  // 后执行，覆盖 SFH
```

**后果**：两者 modData key 不同（`SmartFilteredHopper/Capacity` vs `BiggerAutoGrabber/Capacity`），后加载的 mod 生效。但 hopper 是 Chest 类型，不是 `(BC)165`，BAG 不会处理它，所以**实际不会冲突**。

#### 场景 2: ChestsAnywhere 附加的 Overlay

**问题**：ChestsAnywhere 通过 Event tick 检测菜单并附加 Overlay，不直接 patch 目标方法

**后果**：**不会冲突**。ChestsAnywhere 的 Overlay 是附加在现有菜单上的，不修改 `ItemGrabMenu` 内部逻辑。

#### 场景 3: 其他 mod 的 Transpiler 修改 `Chest.addItem`

**问题**：如果某 mod 用 Transpiler 在 `Chest.addItem` 中插入了检查逻辑

```csharp
// 假设某 mod 的 Transpiler
if (items.Count >= 54) return false;  // 修改了容量检查
```

**后果**：可能导致物品无法放入 hopper 箱子。方案 A 不使用 Transpiler，**不受此影响**。

### 7.4 方案 A 的抗冲突设计

| 设计 | 防冲突机制 |
|------|------------|
| **Postfix 而非 Prefix** | `GetActualCapacity` 只修改返回值，不拦截方法执行，其他 mod 的 Prefix 仍正常执行 |
| **Finalizer 而非 Transpiler** | 不修改 IL 指令，避免与 Transpiler 冲突 |
| **容量抑制模式** | `_suppressCapacity` 是 `[ThreadStatic]`，每个线程独立，不受其他 mod 影响 |
| **非侵入式物品转移** | 通过 Prefix 跳过 `ShowMenu()`，但不修改物品转移核心逻辑 |
| **多重检测 FindHopperChest** | 通过 sourceItem、context、heldObject 三种方式查找，确保能找到 hopper chest |

### 7.5 冲突预防建议

| 建议 | 说明 |
|------|------|
| **设置 HarmonyPriority** | 在 `Patch` 时指定 `HarmonyPriority.HigherThanNormal` 或更高，确保执行顺序 |
| **检测共存 mod** | 在 `GameLaunched` 时检测是否有 BAG 等 mod，给用户警告 |
| **提供兼容性选项** | 如检测到冲突，允许用户禁用容量补丁的某些部分 |
| **保留降级路径** | 如果 `FindHopperChest` 失败，回退到 vanilla 行为，不崩溃 |

### 7.6 冲突检测与日志

```csharp
// 在 ItemGrabMenuCtor_Finalizer 中添加检测
if (wasSuppressed && FindHopperChest(__instance) == null) {
    ModState.Log($"[SmartFilteredHopper] Warning: Menu opened but hopper not found. Possible conflict with other mod.");
}
```

当检测到可能的冲突时，记录日志并让用户知道可能有兼容性问题。

## 8. 注意事项

| 风险 | 缓解措施 |
|------|----------|
| 超过 72 格无法访问 | 明确文档说明容量限制 |
| 与其他 UI mod 冲突 | 使用 `HarmonyPriority` 设置高优先级 |
| 多人游戏兼容性 | 未装 mod 的玩家看不到超容物品 |
| 物品转移逻辑遗漏 | 需完整测试放物/取物场景 |
| 目标方法被内联 | 使用 `HarmonyLib.HarmonyNonInlining` 确保补丁生效 |
